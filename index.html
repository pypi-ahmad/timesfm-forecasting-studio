      CHAPTER 01 / 04

      Introduction

      Build a working mental model of time-series foundation models, patched decoding, probabilistic outputs, and zero-shot transfer before touching the runtime.

      1. Time series and forecasting

      A time series is an ordered sequence of measurements. For a univariate series, each value is associated with one timestamp:

      y1:T=(y1,y2,…,yT)

      Forecasting uses the latest C observations as context to estimate the next H observations, the forecast horizon. This repository intentionally performs univariate forecasting: uploaded columns may aid inspection, but exactly one numeric target reaches the model.

      Core forecasting vocabulary
      TermMeaning hereExample
      ObservationOne timestamped target valueHourly demand at 14:00
      ContextHistorical values supplied to the modelPrevious 720 hours
      Forecast horizonNumber of future steps requestedNext 48 hours
      FrequencySpacing of the timestamp gridHourly, daily, month-end
      Point forecastOne estimate per future stepThe q50/median path
      Quantile forecastEstimate at probability level τq10, q50, q90
      ⚠ Ordered does not mean regular. A sorted series with random gaps is still irregular. The application rejects it until the timestamp grid is repaired.

      2. What is a time-series foundation model?

      A foundation model is pretrained on broad data and reused across downstream tasks. A time-series foundation model applies that idea to temporal sequences. TimesFM starts with parameters learned from a large, multi-domain pretraining corpus rather than estimating fresh parameters from only the uploaded series.

      Google describes TimesFM as a patched, decoder-only model for zero-shot forecasting across domains and granularities. This repository uses TimesFM 2.5 200M for PyTorch at a pinned Hugging Face revision. The primary sources are the official repository, research paper, and Google Research overview.

      2.1 Patching

      Passing every scalar as its own transformer token would make long histories expensive. TimesFM groups consecutive observations into input patches. This app mirrors the 32-point input patch and 128-point output patch when rounding compilation sizes.

      Patching is not seasonal aggregation. Values are not averaged into weekly or monthly statistics; their within-patch ordering remains meaningful.

      2.2 Probabilistic outputs

      TimesFM 2.5 can produce a continuous-quantile forecast. Its interface returns ten channels per future step: the mean followed by q10 through q90. The application displays q50 as the point forecast and shades q10–q90.

      ρτ(u)=u(τ−𝟙[u<0])

      These are model-estimated predictive quantiles, not a guarantee that 80% of observations will fall inside q10–q90 for every dataset.

      3. Zero-shot transfer learning

      Let Dpre denote the pretraining collection and L its forecasting objective. Pretraining learns shared parameters θ*. For a previously unseen series z, zero-shot inference keeps θ* fixed:

      z^T+1:T+H=fθ*(zT−C+1:T)
      What changes at each stage?
      StageData usedParameters changed?In this app?
      Foundation pretrainingLarge multi-domain corpusYesNo; performed by Google
      Zero-shot inferenceUser-selected contextNoYes
      Fine-tuningTarget-domain windowsYesNo
      BacktestingHistorical holdoutsNo model change requiredNot automated
      ⚠ Zero-shot is not zero-validation. It removes task-specific training, not the need to compare forecasts against held-out observations and domain baselines.

      4. TimesFM versus established approaches

      ARIMA and Prophet are normally fit per series. DeepAR learns a global autoregressive model from a collection of related series. TimesFM is pretrained broadly and can forecast a new series without target-domain fitting.

      5. What varying frequency means in TimesFM 2.5

      TimesFM was designed to generalize across granularities, but TimesFM 2.5 removed the frequency indicator from its API. In this application frequency validates the datetime grid and generates future timestamps. It is not passed as an hourly, daily, or monthly feature.

      Turn local or remote tables into the exact regular, univariate, ordered context required by the application—without hiding data-quality assumptions.

      1. Ingestion architecture

      All sources become a pandas.DataFrame. Identical timestamp and target validation then applies regardless of where the bytes originated.

      2. Supported file formats

      3. Loading local files

      4. Loading a public URL

      Paste a direct HTTP or HTTPS URL ending in CSV, Parquet, or XLSX. The resolver treats every remote file as untrusted.

      ControlPurpose
      HTTP/S allowlistReject local file and custom schemes
      No embedded credentialsPrevent username/password leakage
      DNS/IP validationBlock private, loopback, link-local, multicast, reserved, and unspecified addresses
      Redirect revalidationValidate every destination up to the redirect limit

      Query strippingAvoid retaining signed query parameters
      ⚠ Compressed expansion remains local. A compact workbook or Parquet file can consume significantly more memory during parsing.

      5. Datetime detection and parsing

      The loader ranks likely datetime columns rather than silently locking a choice. Native datetime columns rank strongly. String/object columns are sampled and considered when a high proportion parses; date-like names improve ranking. Numeric columns are not guessed because integers may be IDs, years, or epochs.

      5.1 Required timestamp invariants

      ProblemExampleRepair
      Invalid textnot-a-dateCorrect or deliberately remove the source row
      Duplicate timestampTwo readings at 09:00Aggregate or select using domain rules
      Missing grid pointHourly data jumps 09:00 to 11:00Reindex hourly and decide target filling
      Mixed timezone semanticsLocal and UTC stringsConvert explicitly to one timezone
      Too few pointsOnly two timestampsSelect a valid frequency manually

      5.2 Frequency inference

      When no frequency is selected, pandas.infer_freq examines the complete index. The app builds an expected date_range and requires exact equality. Calendar-aware offsets such as month-end remain calendar-aware.

      ⚠ A frequency choice is validation, not repair. Selecting daily does not fill missing daily rows; mismatches are rejected.

      6. Target conversion and missing values

      The selected target is converted with pandas.to_numeric(..., errors="coerce"). Non-numeric cells therefore enter the same validation path as explicit nulls.

      6.1 Interpolation mathematics

      For a missing value between finite observations ya and yb, linear interpolation estimates:

      y~k=ya+tk−tatb−ta(yb−ya)
      Missing patternBehaviorRationale
      Leading target gapsTrim and warnNo left anchor; avoid inventing history
      Internal target gapsPreserve, count, then TimesFM interpolatesNeighboring values anchor interpolation
      Trailing target gapRejectForecast origin is unclear
      Entire target missingRejectNo usable signal
      InfinityRejectNot a valid observation
      ⚠ Interpolation is an assumption. Long gaps can fabricate smooth behavior. Measure duration and prefer domain-specific imputation when appropriate.

      10. Worked repair example

      Suppose hourly readings contain two rows at 10:00 and no row at 11:00. Do not merely select hourly frequency. Decide what duplicates mean, aggregate if appropriate, create the full grid, and then impute the missing target.

      Illustrative repair policy
      frame["timestamp"] = pandas.to_datetime(frame["timestamp"], utc=True)
frame["target"] = pandas.to_numeric(frame["target"], errors="coerce")
frame = frame.groupby("timestamp", as_index=False)["target"].mean()
frame = frame.set_index("timestamp").asfreq("h")
frame["target"] = frame["target"].interpolate(method="time")
frame = frame.reset_index()

      Mean aggregation and interpolation are correct only when they match the measurement process.

      CHAPTER 04 / 04

      TimesFM Mastery

      Connect every sidebar control to model mechanics, interpret the distribution correctly, manage resource limits, and design a defensible evaluation.

      1. End-to-end prediction path

      The checkpoint is lazy-loaded and cached as a Streamlit resource. One process owns one model and serializes compilation/inference through a lock.

      2. Context length

      Context length C is the maximum number of recent observations retained. Older values are excluded. A shorter file contributes all usable observations, but at least two are required.

      2.1 Choosing context by cycles

      Data frequencyCandidate contextPatterns represented
      Hourly168One weekly cycle
      Hourly720Roughly one month and several weeks
      Daily365One annual cycle
      Weekly104Two annual cycles
      Monthly60Five annual cycles
      Context too shortContext too long
      Seasonal pattern may be absentOld regimes may dilute current dynamics
      Recent anomalies dominateMore compile time and memory
      Fewer examples of peaks/troughsHigher OOM risk

      The app warns below 32 observations because that is less than one full input patch.

      3. Forecast horizon

      Forecast horizon H is the number of future periods requested, from 1 to 1,024. Its calendar meaning depends on frequency.

      FrequencyHorizonCalendar interpretation
      Hourly24One day
      Hourly168One week
      Daily30Approximately one month
      Weekly13Approximately one quarter
      Month-end12One year

      Long horizons compound uncertainty. A narrow interval is not evidence of calibration until historical coverage is measured.

      4. Dimension matching and compilation buckets

      TimesFM receives a list with one 1-D numeric array per series. Arrays may have different context lengths, but each app batch shares horizon and non-negative settings.

      TimesFM batch contract
      inputs = [
    series_a.astype(numpy.float32),
    series_b.astype(numpy.float32),
]
point, distribution = model.forecast(inputs=inputs, horizon=horizon)

      The application rounds dimensions to patch boundaries:

      Ccompile=32⌈C32⌉Hcompile=128⌈H128⌉

      It then requires Ccompile + Hcompile ≤ 16,384. Since horizon 1 still compiles to 128, the UI caps context at 16,256.

      RequestedCompiledConsequence
      Context 365384Twelve input patches
      Horizon 24128One output patch compiled; 24 returned
      Horizon 129256Crosses an output boundary
      Context 16,256 + horizon 116,256 + 128Exactly reaches the limit
      ⚠ Boundaries matter. A one-step increase can cross a patch boundary, trigger recompilation, and increase memory use.

      5. Frequency and seasonality

      Frequency controls the datetime contract, not a model feature. It validates historical spacing and generates future timestamps. TimesFM receives only the historical target values.

      TimesFM 2.5 removed the earlier frequency indicator. Selecting hourly, daily, weekly, or monthly does not provide a seasonality token. Calendar-aware pandas aliases still preserve future date semantics such as month-end.

      ⚠ Labels cannot create information. Never mark weekly observations as daily to obtain daily-looking output.

      7.1 Quantile interpretation

      q10 and q90 estimate the 10th and 90th predictive percentiles. Interval width is useful, but calibration must be measured empirically:

      coverage^=1N∑i=1N𝟙[yi∈[y^0.1,i,y^0.9,i]]

      Coverage near 0.8 across representative holdouts supports an 80% interval calibration claim; one chart does not.

      12. Evaluation before production

      The app is a simulator, not a backtesting system. A defensible evaluation repeats historical cutoffs:

      MAE=1N∑|yi−y^i|RMSE=1N∑(yi−y^i)2

      Compare at least a last-value baseline and seasonal-naive forecast ŷt = yt−s. Evaluate interval coverage separately from point accuracy.
